## Two Ways to Customize Components

SvelDocs supports both:

- Global renderer overrides for markdown elements (`h1`, `a`, `pre`, etc.)
- Per-page MDX component imports used directly in markdown content

## 1) Global Markdown Element Overrides

Global renderer mapping is in `src/lib/markdown/components/index.ts`.

`astNodeRenderers` maps markdown element names to Svelte components.

Example keys:

- `h1` through `h6`
- `a`
- `pre`
- `code`
- `table`, `thead`, `tbody`, `tr`, `th`, `td`

To customize a default element globally:

1. Create or edit a component in `src/lib/markdown/components/...`
2. Import it in `src/lib/markdown/components/index.ts`
3. Replace the mapping in `astNodeRenderers`

All docs pages will use the new renderer automatically.

## 2) MDX Imports Inside Content Files

You can import components directly in `.mdx` (and supported markdown with import syntax), for example:

```mdx
import Alert from '$ui/alert';

<Alert type="note">This is a custom component in markdown.</Alert>
```

When content is processed:

- Imports are extracted by `src/lib/markdown/mdx-import-utils.ts`
- A manifest of importable modules is generated by `plugins/vite-mdx-component-manifest.ts`
- Components are resolved and rendered in `src/routes/(docs)/[...slug=docs]/+page.svelte`
- Final rendering is handled by `src/lib/markdown/BlueprintRenderer.svelte`

## Component Name Matching

`BlueprintRenderer.svelte` resolves components by:

- Imported local name
- Alias candidates generated from import source names
- Built-in renderer map keys

That means both direct import names and common filename aliases are supported.

## Practical Pattern

Use global overrides for consistent docs styling, and MDX imports for one-off rich components.

- Global override: best for headings, links, code blocks, tables
- MDX import: best for callouts, demos, and interactive content in a specific page
